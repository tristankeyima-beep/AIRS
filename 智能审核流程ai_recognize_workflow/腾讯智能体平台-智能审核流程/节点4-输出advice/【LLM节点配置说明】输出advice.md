# LLM 节点配置说明：输出 advice

参考 DIFY 提示词：

`DIFY工程-智能审核流程/节点5-输出advice/【prompt】输出adviceV2`

## 一、这个节点做什么

根据节点3的合并审核结果，生成一段给审核人员看的简洁结论。

输入来自：

```text
节点3-按逻辑合并审核+吐出推理过程
```

输出建议命名为：

```text
advice: str
```

## 二、输入变量怎么配

腾讯平台这个 LLM 节点建议配置 3 个输入变量。

| 变量名称 | 数据来源 | 类型 | 推荐绑定 |
| --- | --- | --- | --- |
| `finalResult` | 引用 | str | `节点3.finalResult` |
| `ruleResults` | 引用 | [obj] | `节点3.ruleResults` |
| `logicTopology` | 引用 | str | `节点1.logicTopologyStr` |

说明：

- `finalResult` 是唯一的总审核结论，只能是“通过”或“不通过”。
- `ruleResults` 是每条规则的审核明细，用于总结不通过原因。
- `logicTopology` 是规则组合逻辑树，用于识别 AND / OR 分支，避免把未命中的备选路径当成核心失败原因。

## 三、推荐提示词正文

把下面这段放到腾讯 LLM 节点的提示词里。变量位置建议用腾讯平台变量选择器插入。

```text
# 任务
根据智能审核结果，生成一段面向审核人员展示的简洁审核结论。

# 输入
总审核结果：
【在这里用腾讯变量选择器插入：节点3.finalResult】

智能审核结果：
【在这里用腾讯变量选择器插入：节点3.ruleResults】

规则组合逻辑：
【在这里用腾讯变量选择器插入：节点1.logicTopologyStr】

# 输出目标
让审核人员快速看懂：
- 本次审核是否通过
- 若未通过，核心未通过点是什么
- 哪些规则要求未满足

# 总结范围
1. 必须先读取“总审核结果”，它是唯一的总审核结论。
2. 若总审核结果为“通过”，固定输出“本次审核通过。已满足认定规则组合要求。”，不要再总结 ruleResults 中的单条不通过规则。
3. 若总审核结果为“不通过”，再根据 ruleResults 中 ruleResult 为“不通过”的规则进行总结，重点参考 ruleContent、suspicionList.detail、suspicionList.sources。
4. 不要根据 ruleResults 中是否存在单条“不通过”来推断总结果；总结果只能以“总审核结果”为准。
5. 如果“总审核结果”为空或不是“通过/不通过”，不要自行判断通过或不通过，应输出“未获取到总审核结果，无法生成审核结论。”
6. 总结不通过原因时，不要机械罗列所有单条不通过规则。若某条不通过规则属于未命中的备选分支或互斥分支，且已有同类备选规则通过，不要将该备选分支的不通过作为核心未通过原因。
7. 若规则组合逻辑中存在 OR 分支，应优先找“最接近通过”的分支进行总结：该分支中通过规则最多、未通过规则最少。
8. 对于 OR 结构，不要把所有备选分支的不通过规则混在一起总结；优先总结最接近通过分支中仍未满足的必要规则。
9. 如果某个备选分支已经明显不匹配患者诊断类型，而另一个分支已有多项规则通过，应忽略明显不匹配的备选分支。
10. 当 OR 分支中存在一个“更接近通过”的分支时，最终结论只总结该分支中的未通过规则，不要同时总结其他备选分支的不通过原因。
11. 如果某个分支中已有两个或更多规则通过，而另一个分支全部或大部分不通过，应只总结通过更多的分支。
12. 糖尿病 I 型 / II 型这类互斥诊断路径中，如果患者证据明显支持 II 型路径，应忽略 I 型路径的不通过原因，只总结 II 型路径中仍缺失的必要条件。
13. 若总审核结果为“不通过”，但可总结的不通过规则仅属于未命中的备选分支，应输出“本次审核未通过。现有规则组合未满足完整认定条件，请核查必要规则是否缺失或未通过。”
14. 不要复述 reasoningContent。
15. 总结不通过原因时，仅总结真正导致总结果不通过的规则，忽略通过规则和未命中的备选分支规则。

# 表达要求
1. 输出一段完整中文，不分点、不换行、不加标题。
2. 语言简洁、客观、自然，符合审核结论口吻。
3. 优先概括“规则要求 + 未满足原因”。
4. 多条不通过时，合并同类问题，避免逐条机械重复。
5. 如有明确指标或关键事实，可保留最关键的 1 到 2 项。
6. 避免使用“存在一条不通过规则”“根据推理”“分析可见”等生硬表述。
7. 不要输出思考过程，不要输出 Markdown，不要输出 JSON，不要输出 <think>。

# 输出长度
控制在 50-90 字，必要时可略超，但应尽量精炼。

# 输出风格示例
- 本次审核未通过。现有材料虽支持部分诊断信息，但未提供满足长期治疗要求的有效证据。
- 本次审核未通过。材料明确记载患者无糖尿病史，现有证据不支持“二型糖尿病确诊”这一认定要求。
- 本次审核未通过。患者二型糖尿病诊断及二级以上医疗机构条件已有证据支持，但未见糖尿病急性或慢性并发症诊断证据。
- 本次审核通过。已满足认定规则组合要求。

# 现在开始
请直接输出最终结论文本：
```

## 四、输出示例

### 通过

```text
本次审核通过。已满足认定规则组合要求。
```

### 不通过

```text
本次审核未通过。材料明确记载患者无糖尿病史，现有证据不支持“二型糖尿病确诊”这一认定要求。
```

### 未获取到总结果

```text
未获取到总审核结果，无法生成审核结论。
```

## 五、主工作流串联方式

节点顺序建议：

```text
节点1：规则库转可迭代数组
节点2：调用迭代子工作流
节点3：按逻辑合并审核结果
节点4：输出 advice
```

节点4输入绑定：

```text
finalResult = 节点3.finalResult
ruleResults = 节点3.ruleResults
logicTopology = 节点1.logicTopologyStr
```

节点4输出：

```text
advice: str
```

## 六、联调踩坑记录

1. advice 节点需要拿到 `logicTopology`。
   只看 `finalResult + ruleResults` 时，模型容易把 OR 备选路径的不通过原因混在一起总结。

2. `logicTopology` 建议绑定 `节点1.logicTopologyStr`。
   类型选 `str`，和合并审核节点保持一致。

3. OR 分支要总结“最接近通过”的分支。
   例如糖尿病 I 型 / II 型路径中，如果 II 型路径已有多条规则通过，只缺并发症规则，就不要再把 I 型不通过作为核心原因。

4. 最终 advice 不应复述所有不通过规则。
   目标是一句审核结论，优先总结真正导致最终不通过的必要条件。
