# 启动工作流的输入变量配置说明：节点2-迭代子工作流

## 结论：建议配置 3 个顶层变量

这个子工作流是从父流程的“迭代”里拆出来的。每次启动时，只处理 1 条规则，所以入参建议配置成：

```text
iterator_selector: obj，必填，当前这一次迭代的单条规则
suspicion_type_options: str，建议必填，疑点类型选项
material_list: str，必填，患者材料列表 JSON 字符串
```

## 变量配置表

| 变量名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `iterator_selector` | obj | 是 | 当前迭代到的单条规则。来自父流程节点1输出的 `rulesArray` 里的一个元素。 |
| `suspicion_type_options` | str | 建议是 | 逐条认定时给模型选择的疑点类型。 |
| `material_list` | str | 是 | 患者材料列表。建议传 JSON 字符串，不要拆成对象。 |

## `iterator_selector` 子字段配置

| 层级 | 变量名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `iterator_selector` | `experience` | str | 否 | 经验规则。没有就传空字符串。 |
| `iterator_selector` | `ruleCode` | str | 是 | 规则编码，例如 `1002001`。 |
| `iterator_selector` | `ruleContent` | str | 是 | 规则原文，例如 `二型糖尿病确诊`。 |
| `iterator_selector` | `ruleKeywordGuide` | [obj] | 是 | 该规则需要提取的字段列表。没有也建议传空数组 `[]`。 |

## `ruleKeywordGuide` 子字段配置

| 层级 | 变量名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `ruleKeywordGuide[]` | `keywordCode` | str | 是 | 提取项编码。 |
| `ruleKeywordGuide[]` | `keywordContent` | str | 是 | 提取项说明。建议保留这个字段。 |
| `ruleKeywordGuide[]` | `dataType` | str | 是 | 字段类型，例如 `string`、`enum`、`number`、`date`。 |
| `ruleKeywordGuide[]` | `enumOptions` | [str] | 否 | 枚举选项。只有 `dataType=enum` 时需要。 |
| `ruleKeywordGuide[]` | `required` | bool | 否 | 是否必填。 |
| `ruleKeywordGuide[]` | `instruction` | str | 否 | 额外提取说明。没有就传空字符串。 |

## 父流程传参关系

父流程节点1会输出：

```text
rulesArray: Array
  - item: Object
      experience
      ruleCode
      ruleContent
      ruleKeywordGuide
```

迭代子工作流启动时，建议这样映射：

```text
iterator_selector = 当前迭代项，也就是 rulesArray[i]
material_list = 父流程原始 material_list
suspicion_type_options = 固定字符串或父流程变量
```

## 子工作流内部常用取值

后续节点可以这样理解这些变量：

```text
当前规则编码：iterator_selector.ruleCode
当前规则内容：iterator_selector.ruleContent
当前规则经验：iterator_selector.experience
当前规则提取项：iterator_selector.ruleKeywordGuide
患者材料：material_list
疑点类型：suspicion_type_options
```

## 子工作流对外出参怎么设计

腾讯结束节点会要求先建一个外层对象，比如截图里的：

```text
Output: obj
  one_rule_result: [obj]
```

所以这个子工作流的结束节点建议这样配：

```text
Output: obj
  one_rule_result: [obj]，引用 聚合节点.output
```

注意：这里不是 `output[0]`，而是直接绑定聚合节点的 `output`。因为腾讯这里字段类型只能选 `[obj]`，正好和聚合节点输出一致。

### `Output.one_rule_result` 字段层级

```text
Output: Object
  one_rule_result: Array
    - item: Object
        ruleResult: Array
          - item: Object
              ruleCode: String
              ruleContent: String
              ruleResult: String，"通过" 或 "不通过"
              suspicionList: Array，可选，仅不通过时需要
        reasoningContent: String
        extractionList: Array
          - item: Object
              keywordCode: String
              found: Boolean
              results: Array<Object>
```

## 父流程如何接节点3

父流程里，“调用迭代子工作流 / 迭代”节点会执行多次，每次返回一个 `Output` 对象，里面有 `one_rule_result`。

给节点3传参时，建议这样配置：

```text
ruleResults = 迭代节点聚合后的 Output 数组
logicTopology = 节点1输出的 logicTopology
```

## 重要提醒

当前腾讯平台这种结束节点配置可以用，但需要节点3代码支持拆包。已兼容以下几种形态：

```text
{"Output": {"one_rule_result": [...]}}
{"one_rule_result": [...]}
{"output": [...]}
{"ruleResult": [...], "reasoningContent": "...", "extractionList": [...]}
```

## 联调踩坑记录

1. 子工作流结束节点不能只配一个空 `Output`。
   必须在 `Output` 下配置 `one_rule_result: [obj]`，并绑定聚合节点的 `output`。

2. `one_rule_result` 在腾讯里只能按 `[obj]` 输出。
   不要强行取 `output[0]`；直接绑定整个 `output` 数组即可。

3. 父流程传给节点3时，可能会变成外面包一层 `Output.one_rule_result`。
   节点3代码需要支持拆包；如果节点3读不到规则结果，优先检查这个结构。

4. `rawText` 和 `refContent` 不是一个阶段的字段。
   精解阶段使用 `results[].rawText`；逐条认定输出不通过原因时，才把 `rawText` 摘到 `suspicionList.sources[].refContent`。
