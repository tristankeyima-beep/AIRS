# 代码节点出入参说明：单条标准审核结果结构化

对应代码：`单条标准审核结果结构化.py`

## 这个节点做什么

把“逐条认定 LLM”输出的 JSON 文本整理成统一的 `ruleResult` 数组。通过时输出规则结果；不通过时保留并规范化 `suspicionList`。

注意：这个节点处理的是“审核结论阶段”的数据，所以 `suspicionList.sources` 里使用 `refContent` 是正确的。`rawText` 属于前面的“精解阶段”，不在这里作为标准字段使用。

## 入参

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `ruleResult` | String、Object 或 Array | 否 | LLM 输出的审核结果。一般是 JSON 字符串。 |
| `text` | String | 否 | 兼容字段；当 `ruleResult` 为空时读取。 “【逐条认定节点运行错误】审核结果未读取到”|
| `items` | Object | 建议是 | 当前规则对象。腾讯平台里类型要选 `obj`，绑定 `开始.iterator_selector`，用于兜底补充 `ruleCode`、`ruleContent`。 |

### `items` 层级

```text
items: Object
  ruleCode: String
  ruleContent: String
  experience: String，可选
  ruleKeywordGuide: Array<Object>
```

## 腾讯平台里 `items` 怎么配

```text
变量名：items
数据来源：引用
绑定：开始.iterator_selector
类型：obj
```

不要把 `items` 配成 `str`。如果配成字符串，代码节点不会把它当规则对象使用，也就起不到兜底补充 `ruleCode`、`ruleContent` 的作用。

如果你的“逐条认定 LLM”已经稳定输出了 `ruleCode` 和 `ruleContent`，`items` 理论上可以不配；但建议保留，避免模型偶发漏字段。

代码也兼容一种常见异常：如果 LLM 把 `ruleCode` / `ruleContent` 这两个变量名原样输出成字符串，代码会用 `items` 里的真实规则编码和规则内容覆盖。

### `ruleResult` 推荐层级

```text
ruleResult: Object
  ruleCode: String
  ruleContent: String
  ruleResult: String，"通过" 或 "不通过"
  suspicionList: Array，仅不通过时需要
    - item: Object
        suspicionType: String
        detail: String
        sources: Array
          - item: Object
              materialName: String
              materialId: String
              refContent: String，审核结论引用证据，通常由逐条认定 LLM 从精解结果的 rawText 摘录生成
```

## 出参

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `ruleResult` | Array<Object> | 统一后的规则审核结果数组。即使只有 1 条，也会包成数组。 |

## 联调踩坑记录

1. `ruleResult` 来自“逐条认定 LLM”的完整输出。
   在腾讯变量树里优先选 `逐条认定.Output`，不要只选里面的 `ruleResult` 字段，否则会丢 `ruleCode`、`ruleContent` 或 `suspicionList`。

2. `items` 要绑定 `开始.iterator_selector`，类型选 `obj`。
   它用于兜底补 `ruleCode`、`ruleContent`。配成 `str` 时容易失效；代码已兼容 JSON 字符串，但平台配置仍建议 `obj`。

3. 本节点处理的是审核结论阶段。
   `suspicionList.sources[].refContent` 是正确字段，不要改成 `rawText`。
